## Техническое задание на разработку прототипа адаптивной транспортной системы ChameleonFlow (v0.5)

### 1. Назначение системы

Разработать прототип клиент-серверной системы, которая динамически адаптирует параметры сетевых соединений к изменяющимся условиям среды (потери, задержки, нестандартная маршрутизация, различия в политиках операторов связи). Система использует методы машинного обучения для:
- автоматического определения ухудшения качества соединения (Sensing)
- генерации временных паттернов, статистически неотличимых от типового пользовательского трафика (Morpher)
- интеллектуального выбора оптимального транспортного протокола под текущие сетевые условия (Contextual Bandit)

**Ключевая идея:** повысить надёжность и стабильность связи в условиях недетерминированной сетевой среды без изменения содержимого передаваемых данных.

### 2. Объём прототипа

**Включено:**
- Детектор аномального поведения сети (на базе LightGBM)
- LSTM-генератор межпакетных интервалов для сглаживания трафика
- RL-агент (Thompson Sampling) для выбора транспорта: DoH, WebRTC, QUIC
- Централизованное обновление моделей (раз в сутки)

**Исключено на этапе прототипа:**
- GAN (заменён LSTM)
- Федеративное обучение
- Автоматическая реакция на новые сигнатуры (zero-day) – только ручное/плановое обновление

### 3. Архитектура

```
[Клиент] 
  └── Адаптивный прокси
        ├── Сенсор (классификатор состояния канала)
        ├── Модуль сглаживания трафика (LSTM)
        ├── RL-агент выбора протокола
        └── Плагины транспортов (DoH, WebRTC, QUIC)
                │
                ▼
[Сервер] 
  └── Релейный узел + API управления
        ├── Приём анонимных метрик (только агрегаты успешности)
        └── Выдача обновлённых моделей
```

### 4. Детали реализации ML-компонентов

#### 4.1 Сенсор (оценка качества канала)

**Алгоритм:** Градиентный бустинг (LightGBM) – бинарная классификация: «нормальное соединение» vs «соединение с нарушениями» (потери, аномальные задержки, обрывы).

**Признаки (скользящее окно 5 сек):**
- Доля потерянных пакетов
- Вариация RTT (коэффициент вариации)
- Частота повторных передач (или их аналог для UDP)
- Частота неожиданных сбросов соединения

**Обучение:** на смеси трафика из открытых датасетов (ISCX, UNB) и синтетически сгенерированных помех (эмуляция задержек и потерь через `tc netem`).

**Принцип работы:** Сенсор выдаёт вероятность нарушения качества. При превышении порога 0.7 – инициируется смена транспорта.

#### 4.2 Модуль сглаживания трафика (Traffic Morpher)

**Цель:** сделать временные характеристики исходящего трафика (межпакетные интервалы) статистически похожими на эталонный профиль (например, браузерный веб-сёрфинг). Это снижает вероятность нежелательной классификации трафика промежуточными сетевыми устройствами.

**Данные:** Реальные временные ряды межпакетных интервалов (IAT) из сессий Chrome (публичные датасеты или собранные легально).

**Модель:** LSTM (32 ячейки) + Dense(1) с Huber loss.  
Вход: последовательность из 20 предыдущих IAT (нормализованных).  
Выход: предсказание следующего IAT.

**Применение:** Исходный трафик буферизуется, затем отправляется с задержками, предсказанными LSTM. Реальные данные при этом не изменяются – только темп их отправки.

**Технологии:** PyTorch (обучение) → ONNX Runtime (инференс).

#### 4.3 RL-агент выбора транспортного протокола

**Алгоритм:** Thompson Sampling с контекстом (линейная аппроксимация). Используется библиотека Vowpal Wabbit в режиме `--cb_explore_adf`.

**Действия (протоколы):**  
A = {DoH (DNS-over-HTTPS), WebRTC DataChannel, QUIC}

**Контекст (признаки):**
- Идентификатор оператора связи (ISP ID, один из 10 предопределённых)
- Время суток (час, преобразованный sin/cos)
- Тип трафика (видео, веб, голос – определяется по портам/паттернам)
- Усреднённый успех предыдущих попыток для данного контекста

**Награда:** +1 если в течение 5 секунд после начала передачи сенсор не зафиксировал нарушений, −1 в противном случае.

**Принцип:** Агент перед каждым новым соединением получает контекст, запрашивает распределение вероятностей и выбирает протокол. Через 5 секунд получает награду и обновляет модель онлайн.

#### 4.4 Адаптация параметров TLS (вместо «подмены JA3»)

Для улучшения совместимости с промежуточными сетевыми устройствами и предотвращения ложной классификации используется **гибкая конфигурация TLS-рукопожатия**: набор шифров и расширений, аналогичный наиболее распространённым браузерам (например, Chrome 124). Это не подмена идентификатора, а легитимная настройка TLS-стека, разрешённая стандартами.

**Реализация:** библиотека `utls` (Go) или `python-tls-dynamic` с фиксированным профилем.

### 5. Транспортные плагины (нейтральное описание)

- **DoH-туннель:** передача данных через DNS-запросы (тип TXT). Используется как резервный канал с низкой пропускной способностью в условиях, когда прямые протоколы недоступны.
- **WebRTC DataChannel:** протокол на основе UDP, устойчивый к нестандартной маршрутизации.
- **QUIC:** современный протокол с встроенным контролем перегрузки и шифрованием.

Все плагины работают поверх обычных интернет-портов и не нарушают применимые стандарты.

### 6. Обновление моделей (без федеративного обучения)

Сервер раз в 24 часа собирает от клиентов агрегированные анонимные статистики: количество успешных/неудачных сессий для каждой пары (оператор, время суток, тип трафика, протокол). Эти данные не позволяют идентифицировать пользователя. Сервер переобучает модель бандита и выдает новую версию (файл `.vw`). LSTM-модель обновляется реже (раз в неделю) через централизованно подготовленный файл весов.

**Технологии:** FastAPI (сервер), PostgreSQL (хранение агрегатов), клиент скачивает модели по HTTPS.

### 7. Технологический стек (кратко)

| Компонент           | Технология                               |
|---------------------|------------------------------------------|
| Клиент (ядро)       | Python 3.10 + asyncio                    |
| Сенсор              | LightGBM → ONNX                          |
| LSTM-генератор      | PyTorch → ONNX Runtime                   |
| RL-агент            | Vowpal Wabbit (Python binding)           |
| Транспорты          | aioquic, aiortc, DNS-туннель на asyncio  |
| Сервер обновлений   | FastAPI + PostgreSQL + Docker            |

### 8. Критерии успешности прототипа

1. **Функциональность:**  
   - Система переключает транспорт при ухудшении качества (тест: внесение искусственных потерь 30% через `tc netem`).  
   - LSTM-генератор выдаёт интервалы, проходящие тест Колмогорова-Смирнова против реального браузерного трафика (p-value > 0.05).  
   - RL-агент за 20 итераций находит протокол с наименьшими потерями для каждого из 3 тестовых операторов связи (эмуляция).

2. **Производительность:**  
   - Инференс LSTM ≤ 5 мс на пакет.  
   - Решение RL ≤ 1 мс.  
   - Дополнительная задержка от сглаживания ≤ 150 мс.

3. **Безопасность и приватность:**  
   - На сервер не передаются IP-адреса или содержимое трафика.  
   - Все модели подписаны для проверки целостности.
