# ChameleonFlow: конкретный стек и план реализации прототипа

## 1. Принципы для первого прототипа

- Один основной язык рантайма: `Python 3.11`
- Один способ деплоя окружения: `Docker Compose`
- Один формат инференса ML-моделей на клиенте: `ONNX Runtime`
- Один сервер управления: `FastAPI`
- Один основной способ наблюдаемости: структурированные `JSON logs` + `Prometheus`

Это сужает область реализации и снижает риск несовместимости библиотек.

## 2. Рекомендуемый стек технологий

### 2.1 Клиентский рантайм

- Язык: `Python 3.11`
- Асинхронность и сетевой orchestration: `asyncio`
- Конфигурация: `pydantic-settings`
- CLI и запуск агента: `Typer`
- Логирование: `structlog`
- Метрики: `prometheus-client`
- Формат конфигов: `YAML`

### 2.2 ML и data pipeline

- Обучение сенсора: `LightGBM`
- Экспорт сенсора: `skl2onnx` или `onnxmltools` + `onnx`
- Обучение morpher-модели: `PyTorch`
- Инференс morpher и сенсора на клиенте: `onnxruntime`
- Препроцессинг и offline feature engineering: `pandas`, `numpy`, `scikit-learn`
- Статистическая валидация: `scipy`

### 2.3 RL / contextual bandit

- Библиотека: `Vowpal Wabbit` Python bindings
- Формат модели: `.vw`
- Онлайн-обновление на клиенте: локальный state store + периодическая отправка агрегатов

### 2.4 Транспортные плагины

- Базовый интерфейс транспортов: Python abstract plugin layer
- QUIC: `aioquic`
- WebRTC DataChannel: `aiortc`
- Резервный DNS/HTTPS-канал для прототипа: `aiohttp`

Примечание: в прототипе лучше реализовать не “сырой DNS TXT tunnel”, а `HTTPS`-обёртку для управляющего/резервного канала с DoH-совместимой схемой запросов. Это заметно проще в реализации, тестировании и эксплуатации. Если нужен именно DNS TXT, его стоит выделить во вторую итерацию.

### 2.5 TLS-профили

- Если остаёмся в Python-only прототипе: фиксируем безопасный `TLS profile abstraction`, без попытки глубокой кастомизации handshake
- Если нужна реальная браузероподобная настройка ClientHello: выносим этот слой в отдельный `Go sidecar` на `utls`

Решение для v1 прототипа: не делать отдельный Go-компонент, пока не доказана необходимость на тестовом стенде.

### 2.6 Серверная часть

- API: `FastAPI`
- ASGI server: `uvicorn`
- База данных: `PostgreSQL 16`
- ORM / SQL toolkit: `SQLAlchemy 2.x`
- Миграции: `Alembic`
- Фоновое расписание: `APScheduler`
- Хранение артефактов моделей: локальный volume в прототипе, далее `S3-compatible storage`
- Подпись моделей: `Ed25519` через `PyNaCl`

### 2.7 Инфраструктура и тестовый стенд

- Контейнеризация: `Docker`, `Docker Compose`
- Сетевые эмуляции: `tc netem`
- Нагрузочные и интеграционные сценарии: `pytest`, `pytest-asyncio`
- Линтинг и форматирование: `ruff`
- Проверка типов: `mypy`
- CI: `GitHub Actions`

## 4. Конкретный roadmap реализации

### Шаг 0. Зафиксировать технический baseline

Цель: подготовить каркас проекта и убрать неопределённость по окружению.

Сделать:

- Перевести проект на `Python 3.11`
- Настроить монорепо-структуру: `client`, `server`, `ml`, `infra`, `docs`
- Добавить `ruff`, `mypy`, `pytest`, `pytest-asyncio`
- Поднять `docker-compose` со службами: `server`, `postgres`, `prometheus` при необходимости
- Описать `.env.example` и базовые YAML-конфиги

Результат:

- Репозиторий запускается локально одной командой
- Есть базовый CI и единый стиль кода

### Шаг 1. Спроектировать доменную модель и контракты

Цель: не смешивать ML, транспорты и серверный API без явных интерфейсов.

Сделать:

- Описать `TransportPlugin` interface
- Описать `SensorResult`, `BanditContext`, `BanditDecision`, `SessionAggregate`
- Зафиксировать API:
  - `POST /metrics/aggregates`
  - `GET /models/latest`
  - `GET /health`
- Определить формат локального client state
- Определить versioning моделей и схему подписей

Результат:

- Есть стабильные DTO и протоколы взаимодействия

### Шаг 2. Подготовить датасеты и offline pipeline

Цель: получить воспроизводимый процесс обучения.

Сделать:

- Собрать датасеты для сенсора
- Подготовить сценарии деградации сети через `tc netem`
- Собрать IAT-выборки для browser-like профиля
- Построить единый pipeline feature extraction
- Сохранить train/val/test splits и metadata

Результат:

- Данные и скрипты обучения воспроизводимы

### Шаг 3. Реализовать сенсор качества канала

Цель: первым закрыть механизм детекции ухудшения.

Сделать:

- Написать extractor признаков по окну 5 секунд
- Обучить `LightGBM` бинарный классификатор
- Подобрать порог срабатывания, стартово `0.7`
- Экспортировать модель в `ONNX`
- Подключить инференс в клиенте
- Добавить unit и replay-based тесты

Результат:

- Клиент умеет детектировать деградацию и выдавать probability score

### Шаг 4. Реализовать morpher-модуль

Цель: управлять интервалами отправки без изменения payload.

Сделать:

- Собрать датасет IAT
- Обучить `LSTM(32)` + `Dense(1)` с `Huber loss`
- Экспортировать модель в `ONNX`
- Реализовать буфер исходящих чанков
- Реализовать планировщик отправки по предсказанным задержкам
- Ограничить максимальную добавочную задержку `<= 150 ms`

Результат:

- Исходящий поток может отправляться по предсказанным интервалам

### Шаг 5. Реализовать bandit-агент выбора транспорта

Цель: научить систему выбирать лучший транспорт по контексту.

Сделать:

- Формализовать контекстные признаки
- Обернуть `Vowpal Wabbit` в локальный Python service
- Реализовать:
  - выбор действия перед началом сессии
  - расчёт reward через 5 секунд
  - локальное онлайн-обновление
- Добавить fallback-логику, если модель отсутствует или повреждена

Результат:

- Перед каждым соединением клиент выбирает транспорт осмысленно, а не по round-robin

### Шаг 6. Реализовать транспортный слой

Цель: дать bandit-агенту реальные альтернативы.

Порядок реализации:

1. `QUIC` как основной быстрый транспорт
2. `WebRTC DataChannel` как UDP-альтернатива
3. `HTTPS/DoH-compatible fallback` как резервный канал

Сделать:

- Для каждого транспорта реализовать единый async lifecycle:
  - `connect()`
  - `send()`
  - `recv()`
  - `close()`
  - `health()`
- Добавить health probes и timeout policy
- Собрать relay-сервер с поддержкой всех трёх режимов

Результат:

- Клиент умеет переключаться между транспортами без смены верхнего API

### Шаг 7. Собрать сервер управления моделями

Цель: централизованно принимать агрегаты и раздавать актуальные модели.

Сделать:

- Реализовать приём обезличенных агрегатов
- Сохранять агрегаты в `PostgreSQL`
- Реализовать таблицы:
  - `session_aggregates`
  - `model_registry`
  - `model_downloads`
- Реализовать выдачу latest-моделей с подписью и version metadata
- Добавить nightly job переобучения bandit-модели

Результат:

- Сервер управляет жизненным циклом моделей

### Шаг 8. Добавить безопасность и контроль целостности

Цель: исключить незаметную подмену моделей и утечку лишних данных.

Сделать:

- Подписывать артефакты моделей `Ed25519`
- Проверять подпись на клиенте перед активацией модели
- Явно запретить передачу IP, payload и raw packet traces
- Добавить audit-лог на операции публикации модели

Результат:

- Есть минимально достаточная модель доверия для прототипа

### Шаг 9. Интеграционное тестирование на сетевом стенде

Цель: проверить, что компоненты работают как единая система.

Сделать:

- Подготовить 3 профиля “операторов” через `tc netem`
- Проверить автоматическое переключение транспорта при `30%` packet loss
- Прогнать KS-test для morpher против эталонного браузерного профиля
- Проверить, что bandit за `20` итераций стабилизирует выбор лучшего транспорта
- Измерить latency overhead и время инференса

Результат:

- Есть численное подтверждение критериев из ТЗ

### Шаг 10. Подготовить демонстрационный сценарий

Цель: получить понятный способ показать прототип.

Сделать:

- CLI-команды:
  - запуск клиента
  - запуск relay/server
  - загрузка моделей
  - эмуляция деградации
- Добавить dashboard с ключевыми метриками
- Подготовить demo script на 5-7 минут

Результат:

- Прототип можно воспроизводимо продемонстрировать и оценить

## 5. Что сознательно не включать в первый прототип

- Федеративное обучение
- GAN и любые сложные генеративные схемы вместо LSTM
- Глубокую TLS-мимикрию отдельным низкоуровневым стеком
- Полноценный production-grade DNS tunneling
- Kubernetes и сложный multi-node deployment

## 6. Критические риски

- `aiortc` и `aioquic` могут усложнить кроссплатформенную сборку
- `Vowpal Wabbit` иногда неудобен в локальной установке и CI
- Реалистичный IAT-датасет может оказаться сложнее, чем обучение самой LSTM
- Экспорт `LightGBM` в `ONNX` нужно проверить ранним spike-тестом

## 7. Рекомендуемый порядок начала работ

Если начинать прямо сейчас, первый спринт должен быть таким:

1. Перестроить репозиторий под `client/server/ml/infra`
2. Зафиксировать `Python 3.11`, dev tooling и `docker-compose`
3. Сделать контракты API и plugin interface
4. Поднять FastAPI + PostgreSQL
5. Реализовать заглушки трёх транспортов
6. Подключить sensor inference stub и bandit stub

Это даст рабочий skeleton системы, в который потом можно безопасно встраивать ML-модели.
